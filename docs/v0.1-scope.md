# SkillRelay v0.1 Scope

This document defines the first implementation boundary for SkillRelay.

v0.1 is not the full product. It is the smallest coherent version that proves the core architecture:

local registry -> canonical skill model -> source import -> one agent adapter -> status visibility

## 1. v0.1 Goal

Build a working CLI-first local registry that can:
- initialize a local SkillRelay registry
- import a skill from a local file or directory
- store it in a canonical format
- list and inspect registered skills
- track source and status metadata
- export or push a skill through one initial adapter
- validate basic completeness and safety metadata

## 2. Product Boundary

v0.1 must preserve the project contract:
- local-first
- CLI-first
- registry-centered
- adapter-based
- explicit and selective sync only
- no cloud service
- no global shared runtime
- no GUI

## 3. Must Have

### Registry
- Create or initialize a local registry directory.
- Store canonical skill records.
- Store metadata separately enough that status and provenance can be inspected.
- Support listing all registered skills.
- Support showing details for one skill.

### Canonical Skill Model
- Define a first version of the canonical skill schema.
- Store identity, metadata, source, compatibility, content, and status fields.
- Preserve original imported content where possible.
- Do not silently discard source-specific information.

### Source Import
- Import from a local `SKILL.md` file.
- Import from a local directory containing a skill definition.
- Record import origin and timestamp.
- Detect duplicate names and versions.

### CLI
Minimum command families:
- `skillrelay init`
- `skillrelay list`
- `skillrelay info <skill>`
- `skillrelay import <path>` or `skillrelay install <path>`
- `skillrelay status <skill>`
- `skillrelay validate <skill>`
- `skillrelay export <skill> <target>` or first adapter-specific push command

Command names are still design candidates, but v0.1 must expose these workflows.

### Adapter
- Implement one adapter end-to-end.
- Recommended first adapter: Hermes, because Hermes skills are already visible in this environment and have a clear `SKILL.md` structure.
- Adapter must remain isolated from registry core.
- Adapter must support at least export/push from registry to agent-compatible format.
- Pull/import from agent is desirable but may be v0.1.1 if needed.

### Status Visibility
- Show whether a skill is only in the registry or also exported/synced to an agent.
- Show source/origin.
- Show basic validation state.
- Show conflict state if detected.

### Validation
- Check required fields.
- Check readable content.
- Check obvious malformed metadata.
- Flag risky scripts or external commands if visible.

## 4. Should Have

These are valuable in v0.1 but not mandatory if they threaten the minimal end-to-end flow:
- basic `source add/list/remove`
- source enable/disable state
- simple conflict report for same-name skills
- export as portable artifact
- JSON output mode for CLI commands
- dry-run mode for push/export

## 5. Explicitly Out of Scope for v0.1

Do not build these in v0.1:
- hosted cloud registry
- GUI or web UI
- marketplace features
- automatic background sync daemon
- direct publish to all external sources
- broad multi-agent support
- complex semantic conversion between all agents
- remote source crawling beyond simple local path / explicit URL import
- user account system

## 6. v0.1 Acceptance Criteria

v0.1 is acceptable when a user can:

1. Run `skillrelay init` to create a local registry.
2. Import a local skill file or directory.
3. Run `skillrelay list` and see the imported skill.
4. Run `skillrelay info <skill>` and see identity, metadata, origin, and compatibility.
5. Run `skillrelay status <skill>` and see registry/source/agent state.
6. Run `skillrelay validate <skill>` and receive a clear pass/fail/warning result.
7. Export or push the skill to one supported agent format.
8. Re-run status and see that the skill has been exported or synced.

## 7. Implementation Order

1. Define canonical schema.
2. Define registry directory layout.
3. Implement registry init/read/write.
4. Implement import from local file/directory.
5. Implement list/info/status.
6. Implement validation.
7. Implement one adapter.
8. Implement export/push through that adapter.
9. Add minimal tests around registry, import, status, validation, and adapter export.

## 8. Testing Expectations

Every implemented behavior should have tests for:
- normal path
- missing file / missing field
- duplicate skill identity
- malformed metadata
- registry not initialized
- adapter target missing or unavailable

## 9. Agent Execution Rules

Any code agent implementing v0.1 must:
- read `docs/implementation-contract.md` first
- avoid changing product direction
- avoid adding cloud or GUI concepts
- propose unresolved choices instead of silently deciding them
- keep implementation small and testable
- prefer one clean vertical slice over many incomplete features
