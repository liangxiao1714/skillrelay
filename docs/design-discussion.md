# SkillRelay 项目讨论与长期设计草案

日期：2026-05-24
项目正式命名：**SkillRelay**（2026-05-24 敲定）

> 命名说明：
> - 原工作代号为 "SkillMesh"，后经评估弃用（"mesh" 概念被 Service Mesh 等占用，且缺乏视觉意象）。
> - 最终选定 **SkillRelay** —— "Relay"（中继/接力）天然包含"接收 + 传递"的双向动作，完美映射本项目 push / pull / sync / publish 的核心流程。
> - 单数 "Skill" 而非 "Skills" 遵循英语 attributive noun 规则（如 SkillShare、PackageManager），品牌简洁且符合母语者直觉；"skills" 仍可在文档/tagline 中描述类别时使用。
> - CLI 主命令：`skillrelay`；建议短别名：`sr` 或 `relay`。
> - 词族（可在子命令/术语中沿用）：baton（技能包）、runner（agent 适配器）、track（同步通道）、pass / catch / broadcast 等。

## 1. 项目背景

这是一个围绕“agent skills”管理、分发、同步、转换与发布的本机项目构想。目标不是让所有 agent 共享同一套全部技能，而是把本机作为统一的 skills 中心仓库，并通过适配层让不同 agent 按需使用技能。

该项目面向的场景包括：
- 从多个来源发现和导入 skills
- 把技能同步到不同 agent（Hermes、OpenClaw、Claude Code、OpenCode、Codex 等）
- 让 agent 之间通过本机仓库进行 pull / push / sync
- 将 agent 中新产生的技能再发布回外部技能源

## 2. 核心定位

项目更像是：
- 本地 skills registry
- 多 agent skills bridge
- skills sync / publish layer
- skills lifecycle manager

它不是单纯的技能安装器，也不是单纯的技能列表站点，更不是共享的全局技能运行时，而是一个管理 skills 全生命周期的桥接项目。

## 3. 设计原则

- 本机有一个统一的中心仓库
- 各 agent 保持自己的技能体系和目录结构
- 仅按需同步，不强制全量加载
- 支持技能从外部源进入本地，也支持从本地发布回外部源
- 让不同 agent 通过适配器与中心仓库交互
- 项目应可开源，供他人复用、参考和扩展

## 4. 重要概念

### Skill
一个可复用的技能单元，包含用途、元数据、来源、兼容性和内容。

### Source
技能来源。可以是 SkillHub、GitHub 仓库、本地目录、单个 SKILL.md、某个 agent 的已安装技能目录等。

### Registry
本机统一技能仓库，用于保存已纳管的技能和索引。

### Adapter
针对不同 agent 的适配层，用于技能导入、导出、转换和同步。

## 5. 主要功能板块

### 5.1 本地技能仓库
- 在本机维护统一仓库
- 记录技能内容、来源、状态、版本、归属
- 作为所有 skills 的中心存储位置

### 5.2 技能来源管理
- 接入多个来源
- 支持添加、移除、启用、停用来源
- 来源可来自线上平台、本地目录、GitHub 或其他 agent

### 5.3 技能搜索与发现
- 在本地仓库和外部来源中搜索 skills
- 支持按名称、标签、分类、来源发现技能

### 5.4 技能详情查看
- 查看技能的用途、来源、适用 agent、依赖、状态等
- 帮助判断是否值得安装或发布

### 5.5 技能安装与导入
- 从外部来源安装到本地仓库
- 从本地文件或其他 agent 导入技能
- 作为技能归档的统一入口

### 5.6 技能导出与分发
- 将本地技能导出到指定 agent 或指定目录
- 作为中心仓库向外部分发的出口

### 5.7 多 Agent 适配
- 识别不同 agent 的技能格式与加载习惯
- 对技能进行必要的转换或包装
- 让技能跨 agent 复用

### 5.8 技能拉取与推送
- 从中心仓库拉到指定 agent
- 从 agent 回推技能到中心仓库
- 支持选择性同步

### 5.9 技能同步
- 维护中心仓库与各 agent 的状态一致性
- 只同步需要的技能
- 支持手动或半自动同步

### 5.10 技能转换
- 在通用格式与各 agent 格式之间转换
- 尽量保留技能核心内容
- 标记不可完全转换的部分

### 5.11 技能校验与健康检查
- 检查技能是否完整、规范、可用
- 检查依赖、权限、兼容性和源可达性

### 5.12 技能状态与可用性管理
- 让用户知道技能当前在哪、是否已同步、是否可用
- 追踪技能在不同 agent 中的分布状态

### 5.13 技能版本与变更管理
- 记录版本、更新来源和变更历史
- 便于追踪技能演化

### 5.14 技能冲突处理
- 处理同名、不同版本、不同来源之间的冲突
- 维护主版本或主来源的概念

### 5.15 技能可信度与安全管理
- 标记来源可信度
- 检查风险脚本、外部依赖和潜在危险行为

### 5.16 项目配置管理
- 管理仓库位置、来源列表、适配规则、同步策略等配置

### 5.17 可扩展的适配器与来源机制
- 允许未来增加新的 agent 适配器
- 允许未来增加新的来源类型

### 5.18 开放式生态能力
- 作为开源项目让他人复用
- 支持社区扩展来源、适配器和技能格式

### 5.19 技能发布与共享
- 支持把本地技能发布到外部技能源
- 支持把本地 agent 中产出的技能纳入中心仓库后再发布
- 发布可分为两层：先准备可分发的技能制品，再在目标来源支持时直接推送
- 让项目不仅能消费技能，也能生产和分发技能

## 6. 项目形态建议

- 项目应以 CLI 为主
- 仓库应保存通用技能格式和元数据
- 各 agent 通过适配器接入
- 本地仓库作为中心节点
- 可逐步开源并形成生态

## 7. 讨论结论

当前这个项目的方向已经不是简单的“技能安装器”，而是一个覆盖技能发现、导入、仓库管理、跨 agent 分发、转换、同步、发布的完整系统。

后续开发时，可以继续围绕：
- 需求梳理
- 功能优先级
- 项目命名
- 数据结构设计
- CLI 设计
- 适配器设计
- 测试与发布流程
逐步推进。
